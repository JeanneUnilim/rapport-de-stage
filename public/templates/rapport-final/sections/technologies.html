A — PrimeNG
Boutons, tooltips, états conditionnels et éditeur rich-text
PrimeNG est une bibliothèque de composants Angular prêts à l'emploi. Plutôt que de coder un bouton en HTML brut, on utilise <p-button> qui gère automatiquement le style, les états disabled, les icônes et les animations. Le tooltip est une petite bulle d'aide qui s'affiche au survol d'un bouton : il change dynamiquement selon pourquoi le bouton est désactivé, pour guider l'utilisateur.

PrimeNG propose également p-editor, un éditeur rich-text intégré basé sur la librairie Quill. Il est utilisé dans le dialog d'envoi de mail libre aux inscrits : l'agent CIN peut ainsi rédiger un message mis en forme (gras, italique, listes, liens...) sans avoir à écrire de HTML.

B — Tailwind CSS
Mise en page sans écrire de CSS
Tailwind CSS est un framework utilitaire : au lieu d'écrire du CSS dans un fichier séparé, on applique des classes prédéfinies directement sur les éléments HTML. flex active flexbox, gap-2 ajoute un espace, text-xs réduit la taille. Le préfixe sm: applique le style uniquement sur les grands écrans, ce qui est la base du responsive (adaptation de l'interface à la taille de l'écran).

D — Tables PrimeNG, export CSV & fiche de pointage PDF
p-table, Blob API et jsPDF
La table PrimeNG (p-table) gère automatiquement le tri par colonne, la pagination et la recherche globale.

L'export CSV des activités et des inscrits est généré côté front-end sans librairie externe : on construit le contenu texte ligne par ligne, puis on utilise l'API native du navigateur (Blob + URL.createObjectURL) pour créer un fichier téléchargeable à la volée.

La fiche de pointage est un PDF généré avec jsPDF et son plugin jsPDF-AutoTable. Elle inclut le logo CIN, les informations de l'activité, un tableau des participants avec cases à cocher, et un récapitulatif adultes/enfants en bas de page. Initialement, PDFKit avait été envisagé pour cette tâche. Cette librairie fonctionne côté serveur : elle aurait nécessité d'envoyer toutes les données au back-end pour générer le PDF, puis de le renvoyer au navigateur. jsPDF génère le PDF directement dans le navigateur à partir des données déjà disponibles, sans aller-retour inutile avec le serveur.
localhost:4200/activites — table paginée
export.service.ts — genererCSV() + genererFichePointage()
// Export CSV : API native du navigateur.
// Pas de librairie externe — on crée
// un fichier texte formaté puis on
// déclenche son téléchargement.

genererCSV(entetes, lignes, nom) {
  const contenu = [entetes, ...lignes]
    .map(l => l.join(';'))
    .join('\n')

  // Blob = fichier en mémoire
  const blob = new Blob([contenu],
    { type: 'text/csv;charset=utf-8;' })
  const url = URL.createObjectURL(blob)

  // Lien invisible → clic → téléchargement
  const a = document.createElement('a')
  a.href = url
  a.download = `${nom}.csv`
  a.click()
  URL.revokeObjectURL(url)
}

// Fiche de pointage : jsPDF + AutoTable.
// Génère un vrai PDF avec tableau,
// logo et cases à cocher.

genererFichePointage(activite) {
  const doc = new jsPDF()
  doc.addImage(LOGO_CIN, 'PNG', 14, 10, 50, 18)
  autoTable(doc, {
    head: [['#', 'NOM', 'PRÉNOM', 'ÂGE', '']],
    body: participants.map((p, i) =>
      [i+1, p.nom, p.prenom, age, '']),
    // La dernière colonne affiche
    // une case à cocher dessinée en PDF
  })
  doc.save(`Fiche_${activite.titre}.pdf`)
}
E — FullCalendar
Configuration de l'agenda interactif
FullCalendar est une librairie spécialisée dans l'affichage d'agendas. On lui passe un objet calendarOptions qui définit tout : les plugins activés (mois, semaine, liste, interaction), la langue française, les vues disponibles, et les fonctions à appeler lors d'un clic ou d'un drag & drop.

Chaque activité est représentée par un événement (un objet JavaScript contenant un identifiant, un titre, une date et des propriétés étendues extendedProps) pour les données métier comme l'animateur, le nombre de places ou le statut.
localhost:4200 — Calendrier FullCalendar
calendar.ts — configuration FullCalendar
// calendarOptions : tout le comportement
// du calendrier est défini ici.

calendarOptions: CalendarOptions = {

  // Plugins = fonctionnalités activées
  plugins: [dayGridPlugin, timeGridPlugin,
    interactionPlugin, listPlugin],

  locale:      frLocale,
  initialView: 'dayGridMonth',

  headerToolbar: {
    left:   'prev,next today',
    center: 'title',
    right:  'dayGridMonth,timeGridWeek,listWeek'
  },

  editable:     true, // active le drag & drop
  dayMaxEvents: 2,    // max 2 par case

  // Clic → ouvre le panneau de détail
  eventClick: (arg) => {
    const activite = this.activites
      .find(a => a._id === arg.event.id)
    if (activite)
      this.detailService.ouvrir(activite)
  },

  // Drag & drop → mise à jour en BDD
  eventDrop: (arg) => {
    this.activiteService.update(
      arg.event.id,
      { date_debut: arg.event.start }
    )
  }
}
F — Mongoose & MongoDB
Modélisation NoSQL et dénormalisation
Mongoose est une librairie Node.js qui permet de définir des schémas pour MongoDB (une base de données NoSQL qui stocke les données sous forme de documents JSON plutôt que de tables comme en SQL).

Un choix architectural important a été fait : la dénormalisation. Au lieu de stocker uniquement l'identifiant de la catégorie et de l'animateur dans chaque activité (comme on le ferait en SQL avec une clé étrangère), on embarque directement leurs données (nom, couleur...) dans le document de l'activité. Cela évite des requêtes supplémentaires pour récupérer ces informations et accélère l'affichage du calendrier. En contrepartie, si le nom d'une catégorie change, il faut mettre à jour toutes les activités qui la référencent, un compromis justifié ici par la stabilité des données.
MongoDB Compass — collections
MongoDB Compass — documents activites

MongoDB Compass est l'interface graphique officielle pour visualiser et explorer les données MongoDB. On y voit les 4 collections du projet et le détail d'un document activité avec tous ses champs.

activite.model.js — schéma Mongoose
// Schema : définit la structure des
// documents dans la collection MongoDB.
// required: true = champ obligatoire.

const ActiviteSchema = new Schema({
  titre:       { type: String, required: true },
  date_debut: { type: Date,   required: true },
  statut: {
    type:    String,
    enum:    ['publiee', 'annulee'],
    default: 'publiee'
  },

  // Dénormalisation : on embarque les
  // données de catégorie et animateur
  // directement dans l'activité.
  // Pas besoin de jointure → plus rapide.

  categorie: {
    _id:     ObjectId,
    nom:     String,
    couleur: String   // utilisée pour colorier
  },               // les blocs du calendrier

  animateur: {
    _id:    ObjectId,
    nom:    String,
    prenom: String
  }
})

// Model : interface pour interroger
// la collection "activites" en BDD.
export default mongoose.model(
  'Activite', ActiviteSchema
)
G — Express 5 & API REST
Routes, controllers et middlewares
Express est le framework back-end Node.js utilisé pour construire l'API REST. Une API REST est une interface qui expose des points d'accès (des routes) que le front-end Angular appelle via HTTP pour lire, créer, modifier ou supprimer des données, sans jamais accéder directement à la base de données.

Le projet utilise Express 5, la dernière version majeure, qui apporte notamment la gestion native des erreurs asynchrones. Parmi les middlewares configurés (des fonctions qui s'exécutent entre la réception d'une requête et l'envoi de la réponse) figure le rate limiting : un mécanisme qui limite le nombre de requêtes autorisées sur une période donnée pour protéger le serveur contre les abus et les attaques automatisées. La limite a d'ailleurs dû être augmentée en cours de développement, car certaines pages qui chargent beaucoup de données simultanément dépassaient rapidement le seuil initial.

Le package express-mongo-sanitize avait initialement été prévu pour nettoyer automatiquement les données reçues et éviter les injections malveillantes dans MongoDB. Il a dû être retiré car incompatible avec Express 5 : les validations ont donc été intégrées directement dans les schémas Mongoose.

H — Signals Angular
Communication réactive entre composants
Les Signals sont un mécanisme introduit dans Angular 16 pour gérer l'état réactif de manière simple et performante. Un signal est une valeur observable : quand elle change, tous les composants qui la "lisent" se mettent à jour automatiquement, sans avoir besoin de passer des données manuellement de composant en composant.

Dans le projet, le service partagé ActiviteFormService utilise des signals pour coordonner le calendrier, la table des activités et le formulaire. Par exemple, quand une activité est créée ou modifiée, le signal activiteModifiee est incrémenté, ce qui déclenche automatiquement le rechargement du calendrier et de la table, sans aucune communication directe entre ces composants.
activite-form.service.ts — Signals
// signal() : crée une valeur réactive.
// Quand elle change, Angular met à jour
// automatiquement tous les composants
// qui la lisent.

@Injectable({ providedIn: 'root' })
export class ActiviteFormService {

  // Signal booléen : le formulaire
  // est-il visible ?
  dialogVisible = signal(false)

  // Compteur : s'incrémente à chaque
  // modification → déclenche un rechargement
  // dans tous les composants abonnés.
  activiteModifiee = signal(0)

  notifierModification() {
    this.activiteModifiee.update(n => n + 1)
  }
}

// Dans calendar.ts : on écoute le signal
// avec effect() → rechargement auto.

effect(() => {
  this.formService.activiteModifiee()
  this.chargerActivites()
})
I — Multer
Upload et validation des photos d'activités
Multer est un middleware Express dédié à la gestion des uploads de fichiers. Dans le back-office CIN, il est utilisé pour permettre aux agents de joindre une photo à chaque activité.

Multer prend en charge trois responsabilités : définir où stocker le fichier (dans le dossier uploads/activites/ avec un nom unique basé sur un timestamp), filtrer les formats acceptés (uniquement JPEG, PNG et WebP), et limiter la taille maximale (2 Mo par fichier). Ces vérifications sont faites côté serveur, indépendamment du front-end, ce qui garantit qu'aucun fichier non conforme ne peut être enregistré.
upload.middleware.js — configuration Multer
import multer from 'multer'

// Où stocker le fichier et sous quel nom.
const storage = multer.diskStorage({
  destination: (req, file, cb) =>
    cb(null, './uploads/activites'),

  filename: (req, file, cb) => {
    // Nom unique : activite-[timestamp].ext
    // Évite les collisions de noms.
    const ext = path.extname(file.originalname)
    cb(null, `activite-${Date.now()}${ext}`)
  }
})

// Filtrer les formats acceptés.
const fileFilter = (req, file, cb) => {
  const allowed = [
    'image/jpeg',
    'image/png',
    'image/webp'
  ]
  allowed.includes(file.mimetype)
    ? cb(null, true)
    : cb(new Error('Format non supporté'))
}

export const upload = multer({
  storage,
  fileFilter,
  limits: {
    // 2 Mo maximum par fichier
    fileSize: 2 * 1024 * 1024
  }
})
